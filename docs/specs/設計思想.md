# Design Philosophy (設計思想)
**Authored by goryugo**

LLR が守るべき根本的な考え方。

## 1. "Note is Definition" (ノートこそが定義である)
- ルーチンとは「独立したMarkdownノート」である。
- **識別規則**: 以下のみを「ルーチン（リピート対象）」とみなす。
    - **`routine/` フォルダ内** にあるノートへのリンク。
- それ以外のノートへのリンク（例: 仕事ノート）は、単なる「手順の参照（単発タスク）」として扱う。

## 2. "Link-Driven Estimation" (リンク駆動見積もり)
- リンクを書けば、その「実体」から見積もり時間を取得し、自動で表示する。
- ユーザーが見積もりを入力する手間を最小化する。

## 3. "The Dispatcher" (動的紐付け)
- タスク行に書かれた「やりたいこと」と、記録上の「実体（ルーチン）」を柔軟に結びつける。
- `@` 記法による明示的な紐付けにより、ノートのリネームや移動に強い構造を作る。

## 4. "Single File, Infinite Log" (不変の場所、無限の記録)
- ユーザーは毎日、自分のデイリーノート/作業ノートを思考の定位置として使い続ける。
- 日付が変わると自動的にアーカイブされ、リセットされる。
- これにより、ユーザーの「思考の定位置」を確保する。

## 5. "Large Scale Development Protocol" (大規模開発のドキュメント文化)
- プログラム自体の開発だけでなく、ドキュメントの整備・更新を実装と同時並行で、かつ詳細に行う。
- 仕様（仕様書）、設計（アーキテクチャ）、現状（実装サマリー）、計画（ロードマップ）を常に最新の状態に保つ。
- これにより、コードの迷子を防ぎ、プロジェクトの継続性と品質を担保する。

## 6. "Action-Bound Correction" (補正は LLR の意味ある操作にぶら下げる)
- LLR は、ユーザーの自由な Markdown 編集を常時監視して勝手に「正しそうな形」へ戻しすぎない。
- 自動補正や正規化は、原則として **LLR 自身が意味を理解している操作の直後** にだけ走らせる。
- 典型例:
  - `Toggle Task`
  - チェックボックス短押し / 長押し
  - `Adjust Time`
  - `Reset Task`
- これは「正しさ」よりもまず **編集中の主導権はユーザーにある** とみなすための方針である。
- 逆に、LLR が自分で書き換えた結果として整合性が崩れうる箇所は、同じ操作の流れの中で最小限に補正してよい。
- 将来の補正系機能でも、この原則を優先する:
  - 一般編集への広い常時監視より
  - LLR のコマンド / ジェスチャ / 明示操作に結びついた局所補正を優先する
- 実装では、こうした補正を **post action pass** として束ね、1回の LLR action の後に現在ノート全体を見直してもよい。
  - ただし、その発火条件はあくまで **LLR action 起点** に限定する。

## 7. "@ Marker Protocol" (`@` は未処理、`→` は処理済み)
- LLR の本文マーカーは、原則として **`@...` を未処理、`→...` を処理済み** とみなす。
- LLR が読むのは未処理の `@...` だけであり、処理済みの `→...` は再実行対象にしない。
- 成功したマーカーは、意味に応じて処理済み表記へ正規化する。
  - 例:
    - `@0501` -> `→2026-05-01`
    - `@done` -> `→done`
- この規則により:
  - 同じ指示を二重適用しにくくする
  - 「まだ効いていない指示」と「もう反映済みの履歴」を本文上で見分けやすくする
  - 将来の `@...` 系機能を同じ作法で増やしやすくする
- `@ marker` も、発火タイミングは **Action-Bound Correction** に従う。
  - つまり、自由編集そのものではなく、LLR action の post action pass で読む。

## 8. "Spec-First Rule" (仕様書ファースト)
**必ず守ること**: 新機能や挙動変更を実装する前に、必ず対応する仕様書（`docs/specs/` 内の `.md` ファイル）を先に更新・作成する。
- コードは仕様書から生まれる。仕様書のない実装は行わない。
- 仕様書の更新が承認された後にのみ、実装（コードの変更）に着手する。
- これは「手戻り防止」と「後から読んで理解できるプロジェクト」を実現するための最重要ルール。
