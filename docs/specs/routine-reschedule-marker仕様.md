# routine reschedule marker 仕様

## この文書の役割
- **対象**: task 行に書く `@日付` による routine の先送り
- **目的**: 完了/skip とは独立して、routine の `next_due` を明示的に未来日へ変更する
- **Status**: Implemented

## 背景
- `@done` は「前倒しで見えている回を閉じる」機能であり、意図は start_before に強く結びつく。
- 一方で、実運用では以下のような「次回予定そのものを人間判断で後ろへ動かしたい」ケースがある。
  - 花粉症シーズン終了後は、薬タスクを次の春まで出したくない
  - 毎日の routine だが、今週だけ忙しいので来週まで出したくない
- この用途は、完了状態ではなく **next_due の明示変更** として扱うほうが自然。

## 基本方針
- `@日付` は **未処理の先送り指示** とみなす。
- LLR は自由な Markdown 編集そのものには反応せず、**LLR 自身の意味ある action** の流れでだけこの指示を読む。
- 具体的には、LLR action の直後に **現在ページ全体** を走査し、処理可能な `@日付` をまとめて読む。
- この全体走査は、LLR 内部では post action pass のひとつとして扱う。
- 明示コマンド `Reschedule Routine` も同じく **現在ページ全体** を対象にする。
- 成功したら task 行の `@日付` は **処理済み記号** `→YYYY-MM-DD` に書き換える。
- この書き換え規則は [設計思想](/Users/goryugo/GitHub/llr/docs/specs/設計思想.md) の **@ Marker Protocol** に従う。

## 設計原則との関係
- この機能は [設計思想](/Users/goryugo/GitHub/llr/docs/specs/設計思想.md) の **Action-Bound Correction** に従う。
- つまり:
  - ファイルを手で編集しただけでは発火しない
  - 外部同期や一時的な編集中状態にも勝手に反応しない
  - LLR の checkbox 操作 / Toggle / Skip / Retro Complete / 明示コマンドの流れにだけぶら下げる

## 対象
- routine link を含む **チェックボックス task 行** または `- skip:` 行のみ対象。
- 対応する routine note が **1つだけ** 解決できる行で使う。
- checkbox 状態 (`[ ]` / `[/]` / `[x]`) や `skip` ログかどうかは問わない。
- 通常の本文行やメモ行には `@` が出うるため、反応対象に含めない。

## 記法

### 未処理記号
- `@MMDD`
  - 例: `@0501`
- `@M/D`
  - 例: `@5/1`
- `@M月D日`
  - 例: `@5月1日`
- `＠...`
  - 全角 `＠` も同様に受け付ける
- 追加で、明示年付きの `@YYYY-MM-DD` も受け付ける

### 処理済み記号
- コマンド成功後は `→YYYY-MM-DD` に正規化して残す
- 例:
  - `@0501` -> `→2026-05-01`
  - `＠5月1日` -> `→2026-05-01`

## 日付解釈
- 基準日は runtime today を使う。
- 年なし表記 (`MMDD` / `M/D` / `M月D日`) は、**次に来るその月日** を採用する。
  - 今年分が未来なら今年
  - 今年分が過去または当日なら翌年
- 有効条件:
  - runtime today より未来
  - 1年以内
- 無効条件:
  - 今日以前
  - 1年超先
  - 解釈不能な日付
  - 行内に複数の未処理 `@日付` 候補がある

## 発火タイミング

### 自動発火する LLR action
- checkbox 短押し / 長押しの結果、その行が `- [x]` になったとき
- `Toggle Task` / `Retro Complete Task` などの結果、その行が `- [x]` になったとき
- `Skip Task (Log Only)` の結果、その行が `- skip:` になったとき
- `Adjust Time` / `Reset Task` / `Fix Duration Drift (All)` など、LLR がページを意味的に更新する action の直後

### 走査単位
- 発火時は **現在のデイリーノート全体** を先頭から末尾まで走査する
- 現在行だけではなく、同じページ内に残っている他の pending marker もまとめて処理する

### 明示発火
- `Llr: Reschedule Routine`
- この明示コマンドも **現在ページ全体** を対象にする
- ページ内の `[ ]` / `[/]` / `[x]` / `- skip:` 行のうち、処理可能なものをまとめて反映する

## コマンド動作

### コマンド名
- `Llr: Reschedule Routine`

### 動作
1. 現在ページの各行から未処理の `@日付` を検出する
2. 各行の routine link を解決する
3. 処理可能な行ごとに `next_due` を解釈済み未来日へ更新する
4. 対応する行内の `@日付` を `→YYYY-MM-DD` に置き換える

### 保持するもの
- `repeat` は変更しない
- checkbox 状態は変更しない
- `skip` ログならそのまま `skip` ログとして残す

## 未来日ノート
- 未来日付のデイリーノートでは、このコマンドは実行しない
- 理由:
  - future note は preview-safe を優先する
  - 未来ノートを開いただけで routine state を動かせないようにする既存方針と揃える

## 具体例

### 例1. 未着手のまま来週へ送る
- 入力:
  - `- [ ] [[花粉症の薬]] @0501`
- 実行方法:
  - このページ上で `Reschedule Routine` を実行
- 実行結果:
  - routine note の `next_due = 2026-05-01`
  - 行は `- [ ] [[花粉症の薬]] →2026-05-01`

### 例2. 完了ログに対しても future planning を残す
- 入力:
  - `- [x] 08:00 - 08:05 [[朝の仕込み]] @5/1`
- 実行方法:
  - チェック操作や `Toggle Task` の結果 `- [x]` になった直後に自動処理
  - または `Reschedule Routine` を明示実行
- 実行結果:
  - routine note の `next_due = 2026-05-01`
  - 行は `- [x] 08:00 - 08:05 [[朝の仕込み]] →2026-05-01`

### 例3. skip ログのまま次回日だけ変える
- 入力:
  - `- skip: [[散歩]] ＠5月1日`
- 実行方法:
  - `Skip Task (Log Only)` で `- skip:` 化した直後に自動処理
  - または `Reschedule Routine` を明示実行
- 実行結果:
  - routine note の `next_due = 2026-05-01`
  - 行は `- skip: [[散歩]] →2026-05-01`
