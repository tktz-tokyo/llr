# Today File Format & Recording Rules

ユーザーがデイリーノート/作業ノートにどのように記録されるべきかを定義する仕様書。初期構想では `today.md` 固定を想定していたが、現行の公開説明では一般的な daily note 運用として読む。

## 1. File Definition
- **File Name**: 実装では固定していない。初期構想では `today.md` を想定していた。
- **Location**: Daily note 運用に合わせた作業ノート（Vault Root 固定とは限らない）
- **Role**: ユーザーが常に開きっぱなしにして作業する「作業机」。

## 2. Line Formats (記録フォーマット)

### A. Unstarted Task (未着手)
```markdown
- [ ] [PlannedStart ] [[Task Name]] (Estimate)
```
- 基本的なToDo形式。
- 先頭に開始見込み時刻を置く場合は、本文の一部として保持する。
- 見積もり時間 `(30m)` はここに手動、または自動で書かれる。

### B. Running Task (実行中)
```markdown
- [/] [PlannedStart ] [[Task Name]] HH:mm - (Estimate)
```
- **開始記号**: `- [/]` (Obsidianの標準的な進行中マーク)
- **本文優先**: タスク本文を先に置き、その後ろに実績の開始時刻を置く。
- **実績開始時刻**: 末尾の `HH:mm -` を現在進行中の開始時刻として扱う。
- **開始見込み時刻**: ある場合は本文先頭に保持する。

### C. Completed Task (完了)
```markdown
- [x] [PlannedStart ] [[Task Name]] HH:mm - HH:mm (Est > Act)
```
- **本文優先**: タスク本文を先に置き、その後ろに実績の開始/終了時刻を置く。
- **終了時刻**: 実績開始時刻の右に追記する。
- **所要時間**: `(30m > 25m)` のように、**見積もり > 実績** を記録する。
    - 見積もりがない場合は `(25m)` のみ、または `(? > 25m)`。
- **ログ**: JSONL 等の外部ログ出力は現行仕様では行わない（Markdown 行のみを記録源とする）。

### D. Interrupted Task (中断 / 現行)
```markdown
- [x] [PlannedStart ] [[Task Name]] HH:mm - HH:mm (Est > Act)
- [ ] [PlannedStart ] [[Task Name]] (Remaining Estimate)
```
- **現行仕様**: `Interrupt` は独立ステータスを作らず、**完了ログ (`[x]`) + 未着手複製 (`[ ]`)** として記録する。
- **残り見積り**: 見積りがある場合は `remaining = max(estimate - actual, 0)` で再計算して複製行に付与する。

#### 旧仕様メモ（廃止）
- 以前は `- [-]` を「中断ステータス」として記録していたが、現行仕様では廃止。

## 3. General Rules (記録ルール)

### 1. Strict Parsing & Noise Tolerance
- **タスク行以外は無視する**: ユーザーはタスクの間にメモ、見出し、区切り線を自由に書ける。
- **インデント (計算外)**: 行頭にスペース/タブがある行（インデントされた行）は、**LLR の計算対象外** とする。
    - これらは「サブタスク」「メモ」として扱い、コマンドを実行しても無視される。

### 2. Manual Editing (Respect User Input)
- **Truth at Toggle**: コマンド実行時、エディタ上のテキストが「正 (Truth)」とされる。
- **Use Case**: 
    - 行末の実績開始時刻を `18:34` から `18:40` に手修正して完了すると、`18:40` からの時間が計算される。
    - これにより、書き忘れや修正に柔軟に対応できる。

## 3.5. Quick Input Format (雑入力仕様 v1)

人間が素早く入力するための「雑入力」の解釈ルール。これは**入力時の解釈**に関する仕様であり、保存時の整形書式（`- [ ]`, `- [/]`, `- [x] ...`）とは分けて扱う。

### 目的
- ユーザーは「タスク名 / 開始予定 / 見積」をできるだけ少ないキー入力で書ける。
- 実績（完了時の所要時間など）はプラグインコマンドが生成する。
- 開始予定時刻は入力時に使えるようにする（将来的にはログ保持・分析対象として扱う拡張を検討する）。

### 基本ルール
- 行を空白区切りでトークン化して解釈する（半角/全角スペースを空白として扱う）。
- 採用されなかったトークンを連結したものをタスク名とする。

### トークン解釈
- **4桁の裸数字**: 開始予定時刻として解釈する。
    - 例: `1200` → `12:00`
- **1〜3桁の裸数字**: 見積（分）として解釈する。
    - 例: `10` → `10m`, `120` → `120m`
- **単位付き数字 (`h` / `m`)**: 見積として解釈する。
    - 例: `1.5h`, `10m`

### 複数候補がある場合（前優先）
- 時刻候補が複数ある場合は、最初の1つだけ採用する。
- 見積候補が複数ある場合は、最初の1つだけ採用する。
- 採用されなかった候補トークンはタスク名に残す。

### 妥当性チェック（推奨）
- 4桁時刻は `00:00〜23:59` の範囲のみ有効とする。
- 無効な4桁（例: `2960`）は時刻として採用せず、タスク名に残す。

### 運用方針（曖昧入力）
- 曖昧/紛らわしい入力はユーザーが避ける。
- 例: `AWS 2026` のような行は、意図が曖昧になり得るため書き方を変える（例: `AWS2026`, `AWS 2026年` など）。

### 実装方針メモ（開始予定時刻）
- **v1（現行方針）**: 開始予定時刻は「雑入力の解釈」に利用する入力支援として扱う。
- **将来拡張（実装予定）**: 開始予定時刻を実行ログ/完了ログに保持し、計画と実績の差（遅れ/前倒し）分析に使えるようにする。
- ただし、v1では保存書式の複雑化を避けるため、実行開始時に予定時刻を積極的に保持する仕様にはしない。

#### 完了書式の拡張候補（予定開始時刻の保持）
- 現行では予定開始時刻は `^HH:mm` のような補助記法にせず、本文先頭トークンとしてそのまま保持する。
- 例:
    - `- [x] 18:30 [[🥢晩ごはん]] 18:34 - 19:00 (40m > 26m)`
    - `- [x] [[✍️家族会議]] 19:00 - 19:54 (30m > 54m)`
- 別の補助記法を導入する場合は、現行の本文優先フォーマットとの互換を壊さないことを前提に再検討する。

### 例
- `打ち合わせ 1200`
    - 開始予定=`12:00`, 見積=`なし`, タスク名=`打ち合わせ`
- `1200 打ち合わせ 10`
    - 開始予定=`12:00`, 見積=`10m`, タスク名=`打ち合わせ`
- `読書 10`
    - 開始予定=`なし`, 見積=`10m`, タスク名=`読書`
- `1200 打ち合わせ 1300 30`
    - 開始予定=`12:00`（前優先）
    - 見積=`30m`
    - タスク名=`打ち合わせ 1300`

## 4. Lifecycle (Log Rotation)
**Status: Not Implemented**

- **Event**: `Llr: Start New Day` 実行時。
- **Behavior**:
    1. 現在のデイリーノート/作業ノートの中身を、日時付きファイル名でアーカイブ。
    2. 未完了タスク (`- [ ]`) を新しい当日ノートへ転記（Rollover）。
    3. 今日のルーチンを追加。
