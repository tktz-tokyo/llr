# 未来日付デイリーノートとルーチン基準日ポリシー

## この文書の役割
- **対象**: 未来日付のデイリーノートを作成・閲覧・編集したときに、ルーチンの `next_due` や完了基準日をどう扱うか
- **目的**: 予定確認や先読みのために未来ノートを作っても、ルーチン正本の進行状態が勝手に未来へ進まないようにする
- **優先順位**: 既存の「デイリーノート日付を文脈に使う」利点は残しつつ、状態更新は安全側に倒す

## 問題設定
- 現状は、デイリーノートのファイル名の日付をそのノートの基準日として扱う。
- この方針自体は、過去日の記録や日付文脈に沿った挿入には有効。
- しかし未来日付のデイリーノートでも同じ基準をそのまま使うと、以下の問題が起こる。
  - 未来ノートを開いてルーチンを挿入しただけで、overdue の `next_due` catch-up が未来日まで進む
  - 未来ノート上でタスクを完了すると、「未来の日に完了した」扱いで次回日が決まる
  - 結果として、ユーザーは「予定を見たかっただけ」なのに、実際の運用状態を壊してしまう

## 仕様方針

### 1. 表示系と状態更新系を分離する
- **表示系**:
  - 未来日付のデイリーノートでも、そのノート日付を使って「その日に表示したいルーチン候補」を計算してよい
  - これは planning / preview の文脈として扱う
- **状態更新系**:
  - ルーチン正本の `next_due` 更新、overdue catch-up の永続化、完了基準日の決定は、未来日付ノートを直接の基準にしてはならない
  - 未来ノートは状態遷移のトリガーではなく、原則として表示文脈にとどめる

### 2. 未来日付ノートは preview-safe であるべき
- ユーザーが未来のデイリーノートを作る主目的は、予定の確認、先回りの整理、テンプレート生成である
- そのため未来日付ノートに対しては、**見えてもよいが、勝手に進んではいけない**
- この原則を本仕様の最上位ルールとする

## 用語整理
- **target note date**:
  - 対象デイリーノートのファイル名から解釈された日付
- **runtime today**:
  - 実際にプラグインが動作しているローカル当日
- **state mutation**:
  - ルーチン正本の `next_due` / `repeat` / その他 frontmatter を書き換える処理
- **preview computation**:
  - 正本を書き換えずに、ある target date に表示すべき候補だけを計算する処理

## 必須ルール

### A. ルーチン挿入は非破壊であること
- `Insert Routine` およびデイリーノート自動挿入は、ルーチン候補を計算してノートへ展開してよい
- ただし、その計算の途中で routine 正本の `next_due` を書き換えてはならない
- とくに overdue catch-up は、挿入時には**メモリ上の仮計算**にとどめる

### B. 完了基準日は未来に飛ばさない
- デイリーノート上の完了検知で使う基準日は、以下のように決める
  - target note date が runtime today 以下なら、その日付を使ってよい
  - target note date が runtime today より未来なら、completion base date は runtime today に丸める
- これにより、未来ノート上の操作で「未来に完了した」扱いになることを防ぐ

### C. 未来ノートは state mutation の authority ではない
- 未来ノート上での表示、整形、セクション分け、先行挿入は許可する
- ただし未来ノート上の操作だけを根拠に、ルーチン正本を未来基準で前進させてはならない

## 推奨ルール

### 1. API も役割で分ける
- `fetchDueRoutines(targetDate)` のような表示系 API は、非破壊であることを名前と実装の両方で保証する
- 永続化を伴う catch-up や `next_due` 正規化は、完了処理または明示的な保守処理へ分離する

### 2. notice / debug で意図を見える化する
- 未来ノート上で完了検知が走った場合、内部では today に clamp したことを debug log に残す
- 通常利用では過剰な Notice は不要だが、デバッグ時には判断経路が追えるようにする

### 3. 過去ノート運用は守る
- 過去日のデイリーノートで後追い完了したいケースは正当な運用として残す
- したがって「daily note date を一切使わない」ではなく、「未来日にだけ安全制約をかける」が基本方針

## 代替案と判断

### 案1. 常に runtime today を使う
- 利点:
  - 実装が単純
  - 未来日事故を完全に避けやすい
- 欠点:
  - 過去ノートでの後追い記録が壊れる
  - 日付文脈を読む設計意図が失われる
- 判断:
  - 採用しない

### 案2. 未来ノートでは完了処理を全面禁止する
- 利点:
  - 安全性は高い
- 欠点:
  - ユーザー体験が硬すぎる
  - UI 上は完了できそうに見えるのに反映されない設計になりやすい
- 判断:
  - 初手では採用しない
  - 必要なら将来のオプション候補

### 案3. 表示は target note date、更新は future-clamp
- 利点:
  - 未来ノート preview と過去ノート後追いの両方を守りやすい
  - 既存設計の延長で実装できる
- 欠点:
  - 表示と更新の基準が分かれるため、内部設計を明確にしないと混乱しやすい
- 判断:
  - **採用**

## 実装ポリシー

### フェーズ1
- ルーチン挿入経路から正本更新を外す
- 未来ノートでの completion base date を today に clamp する

### フェーズ2
- 非破壊の preview 用 API と、破壊的な normalize / mutate API を分離する
- テスト名にも `preview-safe` / `state-mutation` の区別を反映する

### フェーズ3
- 必要なら設定項目を追加する
  - 例: 未来ノートでの完了を許可するか
  - 例: 未来ノート完了時に Notice を出すか

## 受け入れ条件
- 未来日付のデイリーノートを作成しても、ルーチン正本の `next_due` はそれだけでは変化しない
- 未来日付のデイリーノートにルーチンを自動挿入しても、overdue catch-up は永続化されない
- 未来日付のデイリーノート上でタスクを完了しても、completion base date は runtime today を超えない
- 過去日付のデイリーノート上での後追い完了は、従来どおりその日付基準で扱える
