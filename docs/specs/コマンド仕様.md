## 1. `Llr: Toggle Task`

### Implementation Reference
- **File**: `src/main.ts`, `service/task-transformer.ts`
- **Primary Method**: `LlrPlugin.handleToggleTask(editor, view)`
- **Logic Engine**: `transformTaskLine(lineText, now)` (Shared with tests)

### Context & Constraints
- **Target File**: 特定のファイル名制限はなく、現在エディタで開いているすべてのMarkdownファイルで動作する。
- **Cursor Focus**: コマンドの種類によってカーソル移動先が異なる。
  - **開始時 (`[ ]` → `[/]`)**: タスク本文の末尾・実際の開始時刻の直前。
    - 理由: 開始直後にその場でメモ（補足）を書き足せるようにするため。
    - 例: `- [/] [[💪プルアップ]] 10:30 -` → `[[💪プルアップ]]` と `10:30` の間
  - **完了時 (`[/]` → `[x]`)**: 完了時刻（終了打刻）の直後。
    - 理由: 完了直後に時刻だけをすぐ手修正できるようにするため。duration や marker の後ろではない。
    - 例: `- [x] [[💪プルアップ]] 10:30 - 11:00 (30m)` → `11:00` の直後
- **Target Line**: 現在エディタでアクティブな行（カーソル位置）。
- **Normalization**: トグル実行時に不要な記号や不整合な時間を自動で「矯正」する。
- **Indentation Check**: 行頭にスペース/タブがある場合は、サブタスクとみなし無視する。

### Behavior Logic

#### 🌟 Hyper-Intelligent Logic (Time-First Parsing)
行内に含まれる「半角数字（HH:mm, XXm, XXh）」を「忖度」して状態を決定する。

| パターン | 現在の状態 | 判定 | アクション |
| :--- | :--- | :--- | :--- |
| **時刻 0個** | 指定なし / `[ ]` | **新規開始** | 現在時刻を本文末尾へ付けて開始 `- [/] Task NOW - ` |
| **時刻 1個** | `/` (実行中) | **通常終了** | 現在時刻で終了打刻 ➔ `[x]` |
| **時刻 1個** | `[ ]` (未着手) | **復元 or 開始** | その時刻を開始見込みとして保持しつつ、実績開始時刻は本文末尾に付けて `/` で開始。 |
| **時刻 1個 + 単位** | `[ ]` (未着手) | **実績補完** | `13:00 読書 5h` ➔ 終了を計算し一撃で完了 `[x]` |
| **時刻 2個+** | `[ ]` (未着手) | **実績修復** | 時刻をソート・計算し、一撃で完了 `[x]` |
| **時刻 2個+** | `[x]` | **リピート** | タスク名を複製し、次の行で **NOW** で開始。 |
| **時刻 2個+** | `/` (実行中) | **予定修了** | 行内の2つ目の時刻を終了時刻として即時完了 `[x]` |

#### Case: Smart Estimate Normalization
- **Input**: `読書 30` または `大掃除 1.5h`
- **Action**: 時刻がなく末尾に数字がある場合、見積もり時間を整形する。
- **Output**: `- [ ] 読書 (30m)` / `- [ ] 大掃除 (90m)`

## 2. Individual Commands (Direct Action)
v0.1.0-pre より導入。トグルの「忖度」に頼らず、ユーザーの意図を強制するためのコマンド群。
これらはすべて `handleToggleTask(..., forceAction)` を経由して実行される。

### 公開状態メモ
- 以下の文書では、実装上存在する内部アクションもあわせて説明する。
- ただし `Interrupt Task` / `Reset Task (Keep Estimate)` / `Fix Duration Drift (All Completed Tasks)` / `Retro Complete Task` は、現行ではユーザー向けコマンド一覧に表示しない。
- 早めに着手したい整理方針として、長押しに割り当てている挙動は「まず内部コマンド相当の単位として定義し、その上でUIから呼ぶ」構成へ寄せる。

### A. `Start Task`
- **Force Action**: `start`
- **Behavior**:
    - **未着手 (`[ ]`)**: 開始する。
    - 本文先頭トークンが `HH:mm` または `HHmm` なら、それを開始見込み時刻として本文に保持する。
    - 実際の開始時刻は、現在時刻を本文末尾へ `HH:mm -` で付与する。
    - 先頭に開始見込み時刻がない場合も、本文はそのままで現在時刻だけを末尾へ付与する。
    - **完了 (`[x]`)**: タスクを複製して開始 (Duplicate & Start)。
    - **実行中 (`[/]`)**: **無視** (Alert: "Action 'start' is not applicable")。

### B. `Stop Task`
- **Force Action**: `complete`
- **Behavior**:
    - **実行中 (`[/]`)**: 本文は維持したまま、末尾の開始時刻 `HH:mm -` を `HH:mm - HH:mm` に拡張して完了 (`[x]`)。
    - **その他**: **無視** (Alert: "Action 'complete' is not applicable")。

### C. `Interrupt Task` (中断)
- **公開状態**: 内部用。ユーザー向けコマンド一覧には表示しない。
- **Force Action**: `interrupt`
- **Behavior**:
    - **実行中 (`[/]`)**: 実行中タスクをその時点で **完了 (`[x]`)** する。
    - 同時に、同じタスク名の **未着手タスク (`[ ]`)** を直下に複製する。
    - 見積りがある場合は、複製行に残り見積り `max(estimate - actual, 0)` を付与。
    - **その他**: **無視**。

### D. `Start Task at Previous Time` (`前の時刻で開始`)
- **Purpose**: 直近の時刻の文脈に合わせて、未着手タスクを開始する。
- **Behavior**:
    - **未着手 (`[ ]`)**: その日の `- [x]` 行全体から終了時刻を見て、最も直近の完了時刻を開始時刻として `[/]` 化する。
    - 将来的に「前回の開始時刻」など別の直近時刻文脈を使う拡張余地はあるが、現状実装の基準は直近完了時刻。
    - 直近完了が見つからない場合は、**現在時刻** で開始する（フォールバック）。
    - **実行中 / 完了**: **無視**（Notice表示）。

### E. `Reset Task (Keep Estimate)`
- **公開状態**: 内部用。ユーザー向けコマンド一覧には表示しない。
- **Purpose**: 誤操作や後編集のため、見積りを優先して保持しつつ、未着手として再解釈可能な形へ戻す。
- **Behavior**:
    - **実行中 (`[/]`)**: `- [ ]` に戻す（本文先頭の開始見込み時刻は保持しつつ、末尾の実績時刻だけを落とす）。
    - **完了 (`[x]`)**: `- [ ]` に戻す（本文先頭の開始見込み時刻は保持しつつ、末尾の実績開始/終了時刻を落とす）。
    - **未着手**: **無視**（Notice表示）。

### F. `Duplicate Task`
- **Force Action**: `duplicate`
- **Behavior**:
    - どの状態であっても、現在の行をコピーして直下に挿入し、**未着手 (`[ ]`)** の形で複製する。
    - 時刻・実績・見積り表記は整理され、本文中心の未着手行になる。
    - 例: `[x] 18:30 [[🥢晩ごはん]] 18:34 - 19:00 (40m > 26m)` ➔ `[ ] 18:30 [[🥢晩ごはん]] (14m)` (New)

### G. `Skip Task (Log Only)`
- **Behavior**:
    - **対象**: デイリーノート内の **未着手 (`[ ]`)** 行、または `- skip:` ログ行。
    - `- [ ]` 行で実行した場合は、先頭だけを `- skip:` に置き換えてログ化する。
    - `- skip:` 行で実行した場合は、先頭だけを `- [ ]` に戻してタスク化する。
    - 時刻・見積り・リンクなど残りの内容はそのまま保持し、サイドバー/ステータスバーのタスク集計対象から外す。
    - 例1: `- [ ] 09:00 [[朝のルーチン]] (15m)` ➔ `- skip: 09:00 [[朝のルーチン]] (15m)`
    - 例2: `- skip: 09:00 [[朝のルーチン]] (15m)` ➔ `- [ ] 09:00 [[朝のルーチン]] (15m)`

### H. `Adjust Time (1m)`
- **Behavior**:
    - 時刻が1つの行（実行中など）は開始時刻を `-1分`。
    - 時刻が2つ以上の行（完了など）は終了時刻を `-1分`。
    - 完了行の duration 表記は再計算する。

### I. `Reschedule Routine`
- **Behavior**:
    - **対象**: 現在ページ内の、routine link を含む **チェックボックス task 行** または `- skip:` 行。
    - ページ全体を走査し、各行の未処理 `@日付` 記法を読み取り、対応する routine note の `next_due` をその未来日に更新する。
    - サポート記法:
      - `@MMDD`（例: `@0501`）
      - `@M/D`（例: `@5/1`）
      - `@M月D日`（例: `@5月1日`）
      - 全角 `＠...`
      - `@YYYY-MM-DD`
    - checkbox 状態や skip 状態は変更しない。
    - 更新成功後は、未処理記号を処理済み記号 `→YYYY-MM-DD` に書き換える。
    - runtime today より未来、かつ 1年以内の日付だけを受け付ける。
    - 未来日のデイリーノートでは実行しない。
    - 通常の本文行・メモ行は対象外とし、`@` を含んでいても反応しない。
    - さらに、この marker 解釈は `Toggle Task`、checkbox click、`Skip Task` などの **LLR action の直後にもページ全体走査で自動実行** される。

### I.1 `@done` marker post-processing
- **Behavior**:
    - `start_before` を持つ routine の完了行で、未処理 `@done` を検出した場合、その回を明示的に閉じる意図として扱う。
    - 有効条件を満たすと、routine の `next_due` は現在の `next_due` を基準に次の1回へ進む。
    - 成功した marker は `→done` に正規化する。
    - 既に完了済みの行へ後から `@done` を追記した場合も、次の **LLR action** の post action pass で現在ページ全体から検出して処理できる。
    - 有効条件を満たさない場合は `@done` のまま残し、通常完了ルールへフォールバックする。

### J. `Fix Duration Drift (All Completed Tasks)`
- **公開状態**: 内部用。ユーザー向けコマンド一覧には表示しない。
- **Behavior**:
    - 現在エディタ内の完了行 (`- [x]`) を走査し、開始/終了時刻から duration を再計算して差分がある行を修正する。
    - 手編集で `(Xm)` がずれた行を一括で正規化できる。

### K. `Retro Complete Task`
- **公開状態**: 内部用。ユーザー向けコマンド一覧には表示しない。
- **Force Action**: `retroComplete`
- **Behavior**:
    - 時刻情報を含む未着手行を、行内の時刻/見積情報から完了行として確定するための明示コマンド。
    - 通常トグルより「後追い完了」を優先して実行したい場面で使う。

## 3. Error Handling (User Feedback)
コマンドが無視された場合（例：実行中でないタスクをStopしようとした）、以下のフィードバックを行う。
- **UI**: 画面右上に `Notice` を表示 ("Command ignored...")
- **Log**: コンソールに理由を出力 (`Command Ignored: Action 'complete' is not applicable...`)

## 3.1 Debug Mode (Trace)
- `Debug mode` 有効時、各コマンド実行の開始/終了/エラーを `Notice` と JSONL に記録する。
- 記録には時刻（秒・ミリ秒）とコマンドIDを含む。
- 目的は通常利用ログではなく、遅延実行・二重発火・無視条件のデバッグ。
- 詳細: [[デバッグモード]]

## 4. Checkbox Interaction (Editor Override)
編集モードでチェックボックスを押した場合、既定トグルの代わりに LLR の専用遷移を実行する。

- 適用範囲: モバイル/デスクトップ共通
- **カーソル挙動**: Section 1 の Cursor Focus と同じルールに従う。
  - `- [ ]` → `- [/]`（開始）: 実際の開始時刻の直前
  - `- [/]` → `- [x]`（完了）: 完了時刻の直後
- 短押し:
  - `- [ ]` -> `- [/]`
  - 開始時刻は `Start Task` と同じ。
  - つまり、本文先頭トークンが `HH:mm` / `HHmm` ならそれを開始見込みとして保持し、実績開始は現在時刻を本文末尾へ付ける。
  - `- [/]` -> `- [x]`
- 長押し（モバイル: 450ms / デスクトップ: 900ms）:
  - `- [ ]` -> `- [/]`（直近完了タスクの終了時刻で開始。見つからなければ現在時刻）
  - `- [/]` -> `- [ ]`（時刻保持）
  - `- [x]` -> `- [ ]`（時刻保持）

## 4.1 Unstarted Planned Start Token Rule
- `- [ ]` 行では、本文先頭トークンだけを開始見込み時刻候補として解釈する。
- `18:00` と `1800` は同じ意味で扱い、内部的には `18:00` に正規化する。
- 解釈した時刻トークンは本文の一部として先頭に保持する。
- 行頭以外に出る `1300` のような時刻風トークンは本文の一部として残す。
- 例:
  - `- [ ] 18:30 [[🥢晩ごはん]] (40m)` -> 開始見込み `18:30`、本文 `18:30 [[🥢晩ごはん]]`
  - `- [ ] 1800 ばんごはん 30m` -> 開始見込み `18:00`、本文 `18:00 ばんごはん`
  - `- [ ] ばんごはん 1800 30m` -> 開始見込みなし、本文 `ばんごはん 1800`
- 長押し成立時は後続 `click` を抑止し、短押しとの二重発火を防止する。
- モバイルの触覚フィードバック:
  - 短押し: 1回振動
  - 長押し: 2連続振動
