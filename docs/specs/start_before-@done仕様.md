# `start_before` と `@done` 仕様メモ

## この文書の役割
- **対象**: `start_before` を持つルーチンに対する `@done` 記号の意味づけ
- **目的**: 実装前に、通常完了と `@done` 完了の境界を明確にする
- **Status**: Implemented

## 背景
- `start_before` により、ルーチンは due より前から表示できる。
- ただし、前倒し表示中にそのタスクを片付けても、通常の完了ルールでは「今見えている回を閉じた」ことにならない場合がある。
- 特に `next_due` がまだ未来のままだと、due-anchor 系では今週ぶんが残り続ける。

## 解きたいこと
- 前倒し表示中の回について、ユーザーが「この回はもう済ませた」と明示できるようにする。
- その明示は常時有効ではなく、**その回の表示期間内だけ**効くようにして誤爆を避ける。

## 提案仕様

### 1. 対象
- `@done` は、`start_before` を持つルーチンに対してのみ意味を持つ。
- `start_before` を持たないルーチンでは、`@done` は単なる文字列として扱う。
- この `@done` も [設計思想](/Users/goryugo/GitHub/llr/docs/specs/設計思想.md) の **@ Marker Protocol** に従い、未処理記号として扱う。

### 2. 発動条件
- タスク本文に `@done` が含まれる。
- 対応するルーチンに `start_before` が設定されている。
- 指示基準日が、その回の表示期間内にある。
- タスクが完了済みかどうかは問わない。

### 3. 表示期間
- `visible_from = next_due - start_before`
- `visible_until = next_due`
- `visible_from <= action_base_date <= visible_until`
- この条件を満たすときだけ `@done` を有効とする。

### 4. 発動時の意味
- 通常の完了日基準再計算ではなく、「今見えている回を済ませた」と解釈する。
- 具体的には、**現在の `next_due` を基準に次の1回へ進める**。
- これにより、前倒し表示で見えている今週ぶんを、due 前に終えた時点で来週ぶんへ送れる。
- この解釈は未着手タスクにも使えるので、「今週はやらない」と決めた時点で `@done` によりその回を閉じられる。

### 5. 非発動時の意味
- 表示期間外では `@done` を無視する。
- その場合は通常の完了ルールを適用する。
- 古い `@done` が本文に残っていても、表示期間外なら誤発火しない。

### 6. テキストの扱い
- 成功した `@done` は **処理済み記号** `→done` に書き換える。
- つまり:
  - 未処理: `@done`
  - 処理済み: `→done`
- 未完了タスクで発火した場合も、タスクの checkbox 状態までは自動変更しない。
- 表示期間外などで発動しなかった場合は `@done` のまま残してよい。
- LLR は未処理の `@done` だけを読む。

### 7. 未来ノートとの関係
- 未来日付ノートでは既存の future-clamp 方針を維持する。
- つまり、表示は未来日文脈でよくても、完了基準日は runtime today を超えて未来へ飛ばさない。
- `@done` 判定に使う日付も、その clamp 後の日付を使う。

## 具体例

### 例1. due 前の前倒し表示で次回へ送る
- `repeat: every week on thu`
- `start_before: 2`
- `next_due: 2026-04-09`
- 2026-04-08 に `@done` 付きで完了

結果:
- 表示期間 `2026-04-07` から `2026-04-09` の内側なので `@done` 有効
- 2026-04-09 の回を消費したものとして、`next_due` は次の木曜へ進む

### 例2. 表示期間外なので通常完了
- `repeat: every week on thu`
- `start_before: 2`
- `next_due: 2026-04-09`
- 2026-04-05 に `@done` 付きで完了

結果:
- 表示期間外なので `@done` は無視
- 通常の完了ルールを適用

## 実装メモ
- `@done` は parser で恒久的な専用構文にするより、LLR action 後の post action pass で読むほうがよい。
- 実装責務としては `main.ts` 側で未処理 `@done` を検出し、`routine-engine` へ `advanceFromDue` 相当の mode を渡し、成功時だけ `→done` へ正規化するのが安全。
