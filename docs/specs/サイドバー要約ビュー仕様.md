## こんな時に見る

- 睡眠タスクをサイドバーから除外したい（`summary_role: sleep`）
- サイドバーで直接タスクを編集したい → 未実装（[[STATE_実装状況サマリー]] §2.B）

## 1. 概要
サイドバーのサマリービューを、デイリーノートの内容を俯瞰するための軽量ビューとして提供する。
現状は「閲覧・ジャンプ中心」の実装とし、編集機能は将来拡張とする。

## 2. 現状仕様（Implemented）

### 2.1 ヘッダー
- 日付表示 `MM-DD (曜)`（曜日は日本語、土日色分け）
- 前日 / `TODAY` / 翌日 ナビゲーション
- 日付ラベルクリックで該当デイリーノートを開く
- メトリクスカード:
  - `EST. TOTAL`: 「未来」に残っている通常タスクの見積り合計
  - `EST. FINISH`: 「未来」の通常タスクだけで見た終了予定時刻
  - `summary_role: sleep` のタスクが「未来」に含まれる場合、`EST. FINISH` に `HH:mm - HH:mm` 形式で睡眠込みの目安時刻を併記（後半小さめ）
  - `summary_role: sleep` は `EST. TOTAL` / `EST. FINISH` の通常計算から除外し、睡眠込みの補助時刻だけに使う
  - `summary_role: sleep` より下に元ノートで並んでいた未完タスクは「今日はやらない」扱いとして、サイドバー表示とヘッダー計算の両方から除外する
  - `summary_role: sleep` が完了済みになった場合、その日の未来タスク表示は全体を閉じる。`sleep` の上下に関係なく、未完タスクはサイドバー未来欄とヘッダー計算から外す

### 2.2 リスト構造
- サイドバーはロジック上、`過去` と `未来` の2つだけで扱う
- 表示順は `過去` -> `未来`
- 同一行の重複表示を抑制（行番号ベース）
- タスク表示は2段構成
  - 上段: `HH:mm - HH:mm` + durationバー
  - 下段: タスク名（`[[wikilink]]` はクリック可能）
- 実行中タスクは薄いボックス背景で強調
- 完了タスクは打ち消し線を使わず、時刻と本文を少し弱いトーンで表示する
- `過去`: 完了タスクを実績時刻順で並べたもの
- `未来`: 未完タスクを、`進行中 -> 進行中より上にある未完 -> 残り` の順で並べたもの
- `未来` のうち、元ノートで `summary_role: sleep` より下にある未完タスクは表示しない
- CSS class は見た目のためだけに `完了 / 実行中 / 取り残し見た目` を使ってよいが、並び順・見出し・時刻計算のロジックには使わない

### 2.3 動的時刻積み上げ (Dynamic Time Stacking)
- `過去`: 実績時刻をそのまま表示する
- `過去` の並び順は単純な `HH:mm` 昇順ではなく、日付判定ロジックと同じ日付境界（現状 `03:00`）で正規化する
  - 例: `23:40` の後に `00:30` の完了 `sleep` が来た場合、`00:30` はその日の末尾側として扱う
- `未来`: 現在時刻を起点に、上から順に時刻を積み上げる
- `未来` の先頭に実行中タスクがある場合:
  - 見積あり -> 開始 + 見積の終了予定を表示
  - 見積なし -> 表示上の終了時刻は現在時刻に寄せる
- `未来` の2件目以降は、直前の終了予定から順に積み上げる
- 未来の長さは制限しない。深夜をまたいでも、そのまま翌時刻として積み上げる
- 未着手行に単一時刻（例: `- [ ] 12:00 [[昼ごはん]] (60m)`）がある場合:
  - チェック記号直後の先頭トークンが `HH:mm` または `HHmm` なら、`HH:mm` に正規化して「予約開始時刻」として扱う
  - 例: `- [ ] 1800 ばんごはん 30m` は `18:00` 開始見込みとして扱う
  - 時刻トークンはタスク本文表示から除外する
  - 現時点では並び順や時刻積み上げの拘束には使わない
  - タスク名の直後に tiny の時計 + 時刻を表示する
  - `displayStartTime` が予約開始時刻を超えた場合のみ、時刻欄を段階的に赤くする
  - 着色は 1分超過から開始し、5分刻みで強くなり、60分超過で上限に達する

### 2.4 セクション区切り（Routine Sections 連動）
- `Routine Sections` 設定（時刻 + ラベル）を使って、リスト中に軽い区切り線 + ラベルを差し込む
  - 例: `---------- 午前 ----------` 相当の視認性を狙った表現
- 判定基準は主に `displayStartTime`（表示時刻）
- セクション内に項目がある場合のみ表示（空セクションは出さない）
- `過去` は、実績時刻順の並びを上から見て見出しを付ける
- `未来` は、時刻積み上げ後の並びを上から見て見出しを付ける
- `過去` と `未来` の見出し判定は独立して行う
- 深夜（`00:xx` など）は、最初の見出しより前でも最後のセクションへ折り返して扱う

### 2.5 時刻警告（Estimate Warning）
- セクション区切りラベルの着色は検討したが、意味がぶれやすく視認性も不安定になりやすいため採用しない
- 通常タスクに対する一律の時刻着色は、現時点では採用しない
- 設定 `Estimate Warning` が ON のとき、残り見積り時間に基づく予定警告表示を有効にする
- 例外として、予約開始時刻付きタスクだけは超過時に段階着色する
- セクション見出しは無色固定とする

### 2.6 自動スクロール（実行中タスク）
- 目的は「今のタスクを見失いにくくする」ことであり、通常閲覧の手動スクロールを奪いすぎないことを優先する
- 強い自動スクロール（上寄せ）:
  - 新規表示時
  - 前日 / `TODAY` / 翌日 の日付切替時
  - 対象日ファイルが切り替わった直後
  - 実行中タスクがあれば、リスト上端から少し余白を残した位置へ合わせる
  - `TODAY` は現在 `requestAutoScrollToRunning()` を使い、上記の強い自動スクロール条件に含まれる
- 弱い自動スクロール（最小限の補正）:
  - 通常の再描画時は、まず前回のスクロール位置を復元する
  - そのうえで、実行中タスクが見切れている場合だけ、表示領域に入る最小量だけ補正する
  - 実行中タスクがすでに快適に見えている場合は動かさない
  - 実行中タスクが切り替わった場合は、通常再描画でも一度だけ位置補正を許可する
- 手動スクロール尊重:
  - ユーザーがリストを手動スクロールした後は、一定時間（現状 15 秒）弱い自動スクロールを抑止する
  - これにより、内容確認中に勝手に「今」へ引き戻されることを避ける
- 実装上の安定化:
  - スクロール位置計算はリストコンテナ基準の相対座標で行い、ヘッダー領域に潜り込みにくくする
  - 起動直後や再描画競合で強い自動スクロールが上書きされにくいよう、短時間の再適用ウィンドウを持つ
- 実行中タスクがない場合は、通常のスクロール位置復元を優先する
- 将来の拡張候補:
  - 実行中タスクがないとき、`TODAY` 押下時だけ未着手先頭項目へスクロールする案は未実装
  - 現状は「実行中がなければスクロール位置復元を優先」が正式仕様

### 2.7 更新モデル
- `metadataCache.on('changed')` による対象日ファイル変更検知
- 1秒デバウンスで再描画（入力中の過剰更新抑制）
- `DocumentFragment` + `replaceChildren` による一括置換でフリッカー低減
- アクティブノート切替時は対象日を再解決して再描画
- アクティブノートが同じ対象日ファイルのままでも、サイドバーへ戻ったタイミングで軽い再同期を許可する

### 2.8 クリック挙動
- タスク行クリック: 該当 Markdown 行へジャンプ（カーソルは行末）
- タスク名内 `[[wikilink]]`: 対応ノートを開く（行ジャンプより優先）
- 日付ラベルクリック: 表示日付のデイリーノートを開く

## 3. デザイン方針（現状）
- Obsidian標準UIに馴染む密度・コントラストを優先
- メトリクスカードは主張を弱める（過度なダッシュボード感を避ける）
- ノート側の見た目は極力いじらず、サイドバー側で必要な視認性を担保する
