## 1. 目的
タスクの完了（`- [x]` への遷移）をトリガーとして、そのタスクの「正本」であるルーチンノートの次回予定日（`next_due`）を自動更新する。

## 2. 設計原則 (Design Principles)
- **標準APIの優先**: 自前での複雑なファイル検索やパースは避け、可能な限り Obsidian の標準 API (`metadataCache`, `fileManager.processFrontMatter` 等) を使用する。これにより、エイリアス対応やファイルの安全な更新を担保する。

## 3. 日本語ショートカット（`repeat`）
- ルーチンの指定には、基本的に YAML の `repeat` を使う。
  - 旧 `schedule` は互換読み込み対象として扱うが、新規記述は `repeat` を推奨。
- `repeat` は短く書けるようにする。内部では `every ...` 形式に正規化して解釈する。

### 3.1. 日数ベース
- `repeat: 1` -> 毎日（`every day`）
- `repeat: 10` -> 10日ごと（`every 10 days`）

### 3.2. 週次（曜日指定）
- `repeat: sun mon` -> 毎週 日・月（`every week on sun mon`）
- `repeat: sun,mon` / `repeat: sun:mon` -> 上と同じ（`,` / `:` 区切り。全角 `：` も可）
- `repeat: 日月火` / `repeat: 月水金` -> 毎週の曜日連結指定
- `repeat: 日 月` / `repeat: 日　月` -> 半角/全角スペース区切り
- `repeat: 毎週月曜` / `repeat: 毎週月曜日` -> 毎週 月曜
- `repeat: 毎週月水金` -> 毎週 月・水・金
- `repeat: 隔週月曜` / `repeat: 隔週月水金` -> 隔週（`every 2 weeks on ...`）

### 3.3. 月次（日付指定）
- `repeat: 毎月1日` / `repeat: 毎月15日` -> 毎月その日
- `repeat: 毎月5,10,15日` -> 毎月 5日・10日・15日（`、` も可）
- `repeat: 毎月末` -> 月末（`every month on last day`）
- `repeat: 毎月末-1日` -> 月末の1日前（`every month on last day -1`）

### 3.4. 月次（第N曜日指定）
- `repeat: 第2土曜日` -> 毎月第2土曜
- `repeat: 第2,3日曜日` -> 毎月第2・第3日曜
- `repeat: 最終土曜日` -> 毎月最終土曜

### 3.5. 例外・詳細指定
- `repeat: every week on mon` のような英語詳細構文もそのまま使用可能
- 位相維持が必要な場合のみ `from due` を使う（例: `every 5 days from due`）

## 4. コア機能：タスクの特定と更新 (Phase 1)
まずは「完了 -> 正本特定 -> 日付更新」を確実に動作させることに集中する。

### 4.0. 前倒し表示（`start_before`）
- `start_before` は `next_due` の**何日前から表示開始するか**を表す補助パラメータ。
- `next_due` の更新ロジック自体は変えない。完了時の再計算は常に `repeat` / `next_due` の規則で行う。
- サポート値:
  - 数値: `5`
  - 数字だけの文字列: `"5"`
  - 英語: `5 day`, `5 days`
  - 日本語: `5日`
- 解釈できない値は無視し、未指定と同じく `0` 日扱いにする。
- 表示判定:
  - `start_before = 0` -> `next_due` 当日のみ表示
  - `start_before = N` -> `next_due - N日` から `next_due` まで表示
  - `rollover: true` を明示した場合は、締切超過後も `next_due` を進めず、その回のまま表示を継続する
  - `rollover` 未指定または `false` の overdue は、次の期日まで catch-up したうえで、その新しい期日を基準に前倒し表示を判定する

#### 4.0.a `@done` による前倒し期間中の明示完了
- **Status**: Implemented
- `@done` は、`start_before` を持つルーチンに対してのみ意味を持つ補助マーカーとする。
- 目的は、前倒し表示で見えている「まだ due ではない回」を、ユーザーが**この回はもう済ませた**と明示したときに次回へ送ること。
- 発動条件:
  - タスク本文に `@done` が含まれる
  - 対応するルーチンに `start_before` がある
  - 指示基準日が、その回の表示期間 `next_due - start_before` から `next_due` の範囲内にある
- 完了状態は必須ではない:
  - `- [ ]` / `- [/]` / `- [x]` のいずれでも、`@done` は「今見えている回を閉じる」指示として扱う
  - つまり `@done` は「完了マーカー」というより、**その回を消費して次回へ送る control marker** である
- 発動時の挙動:
  - 通常の「完了日基準の再計算」ではなく、**現在の `next_due` を基準にその回を1回ぶん消費**して次回日へ進める
  - これにより、`repeat: every week on thu` のような due-anchor 系でも、火曜や水曜の前倒し表示中に「今週ぶんは終えた」と判断して来週へ送れる
- 非発動条件:
  - `start_before` がない
  - 完了基準日が表示期間の外にある
  - `@done` が付いていても、その回の表示期間外なら**通常の完了**として扱う
- テキストの扱い:
  - 未処理の `@done` だけを読む
  - 成功した `@done` は **`→done`** に正規化して残す
  - 未完了タスクで発火した場合も、ステータス自体は自動で完了に変えず、行には `→done` を残す
  - これは `@` を未処理、`→` を処理済みとする **@ Marker Protocol** に従う
- 安全性:
  - `→done` は再処理対象にしないため、同じ指示を重複適用しにくい
  - 表示期間外では `@done` を無視するため、条件外の時点では通常完了へフォールバックできる
  - 未来日付ノートでは、既存の future-clamp 方針を優先し、完了基準日を runtime today より未来に飛ばさない

### 4.1. 人間向けの考え方（重要）
- **基本は「完了した日（今日）基準」**で次回日を決める。
  - これが日常運用の直感に合う（例: 今日やったなら、毎日は明日 / 毎週月曜は次の月曜）。
- ただし、一部のルールは「カレンダー上の周期の位相」を保ちたい。
  - 例: 「5日ごと」を、完了タイミングではなく既存の周期列に沿って進めたいケース。
  - そのため、`repeat` の書き方で意図を分ける。
    - 通常: `every ...`（完了日基準）
    - 例外: `every ... from due`（`next_due` 基準 / 位相維持）

### 4.2. トリガーと詳細判定 (Trigger)
- **完了検知**: `metadataCache.on('changed')` を監視し、ノート内のタスク状態の変化（`- [ ]` から `- [x]` への遷移）を検知する。
- **デバウンス処理**: ファイルパスをキーとして、最新の状態をデバウンスする。通常は遅延により誤操作を吸収する設計だが、**デバッグ段階では 0ms（即時）** を使用中。将来、遅延値を戻す可能性がある。
- **複数リンクへの対応**: 行内に複数の `[[WikiLink]]` がある場合、**そのすべてを抽出**し、該当するすべての正本ファイルを更新対象とする。
- **Undo / 手動編集への対応**: リアクティブ監視により、Undo によるチェック解除や手動でのマーク削除も確実に検知し、不必要な更新を未然に防ぐ。

### 4.3. 正本ファイルの特定 (Resolution)
1. **リンク解決**: `metadataCache.getFirstLinkpathDest` 等を使用し、エイリアス等が含まれていても正しく `TFile` を導き出す。
2. **範囲制限 (Folder Restrictions)**: 
    - 特定のフォルダ（デフォルトでは `routine/`）の直下にあるファイルのみを対象とする。
    - **サブフォルダは対象外**とする（フラットな管理を標準とする）。

### 4.4. 次回予定日の更新ロジック (Update)
1. **情報の取得**: 正本ファイルの YAML から `repeat`（互換で `schedule` も可）と `next_due` を読み取る。
2. **日付の算出**: 
    - リピート規則が判定できた場合: `yaml-parser.ts` に準拠して計算。
    - **通常ルール（デフォルト）**: **「完了した日（今日）」を基準**に再計算する。
      - 例: `every day` は何度完了操作しても「明日」になる（未来へ連続加算しない）。
      - 例: `every week on mon` も、その時点の「今日」から見た次の月曜を計算する。
    - **`@done` 例外ルール**:
      - `start_before` を持つルーチンで、かつ完了基準日がその回の表示期間内にある場合のみ有効。
      - 有効時は「前倒しで見えているその回を済ませた」と解釈し、`next_due` を起点に**次の1回**へ進める。
      - 成功した marker は `→done` に正規化する。
      - 表示期間外では `@done` を無視し、通常ルールまたは `from due` ルールへ戻す。
    - **例外ルール（`from due`）**: `next_due` を基準とする周期（位相維持）。
      - `next_due` が**未来**なら、完了しても **変更しない**。
      - `next_due` が**過去または今日**なら、周期列を保ったまま **「今日より後」で最も近い日**まで進める。
    - **`repeat: none/no` の場合**: 次回予定日を計算せず、既存の `next_due` を**削除**する。
    - **規則未設定時の「親切な補完」**: `repeat` が存在しないノートを完了した場合、`repeat: 1`（毎日）を自動付与した上で翌日の予定を立てる。
    - **判定不能時のフォールバック**: 規則が解釈不能な場合は、一律で「翌日」に更新する（既存メタデータは維持）。
3. **書き込み**: `processFrontMatter` を使用し、既存の YAML の構造を維持して `next_due` 等を書き込む。
    - **YAMLエリアがないノート**: `processFrontMatter` の標準挙動により、必要に応じて自動的に YAML エリアが作成され、フィールドが追加される。

## 5. 検証・ブラッシュアップ項目
基本動作が安定した後、以下のようなケースを順次検証・対応する。
- 存在しないノートへのリンクが含まれるケース。
- エイリアス付きリンクの確実な解決。
- 同じ日に同じタスクを複数回完了させた場合の挙動の最適化。
